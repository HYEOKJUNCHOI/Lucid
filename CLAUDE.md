## 3. 작업 원칙
- 추측 금지. 확인한 내용만 반영한다.
- 최소 변경. 요청 범위를 넘어서는 수정을 하지 않는다.
- 지정하지 않은 파일은 건드리지 않는다.
- 커밋은 반드시 컴파일 에러 없는 상태에서만 한다.
- 판단이 어려우면 무조건 질문한다. 추측으로 진행하지 않는다.
- 답변이 모호하면 즉시 재질문한다. 넘어가지 않는다.
- 지시가 짧거나 디테일이 부족하면 구체화 질문을 먼저 한 후 작업을 시작한다.
- **새 파일(목업·뼈대 포함) 생성 전 반드시 사용자와 상의한다. 목록·구조 제안 → 승인 후 구현.**

## 4. Communication
- **Language:** 모든 설명, 주석, 확인 요청, 계획 보고는 반드시 **한국어**로 작성한다.
- **Method:** 초보자도 이해할 수 있게 쉽게 설명하되, 구조와 로직을 명확히 짚는다.
- **Proactive:** 리스크나 더 나은 대안이 있으면 능동적으로 제안한다.
- **Format:** **[결론/해결책] → [코드] → [상세 설명]** 순서로 답변한다.
- **설명 포맷:**
  1. 코드 블록 먼저
  2. 구분선 `---`
  3. 영어 단어 한국어 발음/뜻 bullet: `* 단어(발음 : 뜻)`
- **확인 멘트:** 파일 수정 전 동의 요청 포함 모든 커뮤니케이션은 한국어로 한다.
  - 예: "BasicInfoDoc.jsx를 수정하려고 합니다. 진행할까요?"
- **🍞 추론 흔적:** 지시가 모호해서 추론했다면 응답 끝에 반드시 남긴다.
  ```
  🍞 추론 흔적: [무엇이 모호했는지] → [어떻게 쓰면 더 명확했을지]
  ```
  - 해석이 2가지 이상이고 확신이 없으면 추론하지 않고 바로 질문한다.

## 5. 레이아웃 변경 규칙

### 5-1. 최소 변경 원칙
- 명시하지 않은 요소의 위치, 크기, 패딩, 정렬은 절대 변경하지 않는다.
- 기존 요소를 임의로 이동하거나 재배치하지 않는다.

### 5-2. 기호 체계

| 기호 | 이름 | 의미 |
|------|------|------|
| `[]` | 앵커 | 이미 있는 요소. 건드리지 않는다. 위치 참조용. |
| `<>` | 포인터 | 삽입 위치. "여기에 넣어라". |
| `{}` | 이동 | 다른 곳에 있는 요소를 여기로 옮긴다. |
| `{+}` | 생성 | 새로 만들어서 삽입한다. |
| `{*}` | 부유 배치 | position + z-index로 띄워서 겹치게 배치한다. |

**조합 규칙:**
- `<{A}>` → A를 여기로 이동
- `<{+A}>` → A를 새로 만들어서 삽입
- `<{A,B}>` → A와 B를 함께 이동
- `<{A,+B}>` → A는 이동, B는 새로 생성하여 함께 삽입

**픽셀 넛지:**
- `{요소} -Npx` → 왼쪽으로 N픽셀 이동
- `{요소} +Npx` → 오른쪽으로 N픽셀 이동

### 5-3. 대규모 재구성 시
1. 즉시 실행하지 않고 변경 범위와 영향받는 요소 목록 먼저 보고한다.
2. 사용자 승인 후에만 실행한다.

## 6. 작업 프로토콜

### 세션 시작 규칙
- CLAUDE.md + HANDOFF.md + PROJECT.md + COMMIT_LOG.md + **TROUBLESHOOTING.md** 를 읽고 시작한다.
- TROUBLESHOOTING.md: 버그 수정·개선 포인트 누적 기록. 같은 실수 반복 방지용.

### 핸드오프 규칙
- 사용자가 **"핸드오프 해"** 라고 지시하거나 작업 완료 시 수행한다.
- 핸드오프 절차:
  1. 현재 작업을 컴파일 에러 없는 상태까지 마무리한다.
  2. **`COMMIT_LOG.md` 업데이트**: 최신 작업 내용을 템플릿에 맞춰 최상단에 추가한다.
  3. `HANDOFF.md` 업데이트: 다음 세션에서 인지해야 할 현재 상태를 기록한다.
  4. `git add` → `commit` → `push` 수행한다.

### 브랜치 전략
```
main
  └── develop
        ├── feature/cc-기능명
        └── fix/cc-버그명
```

| 타입 | 의미 |

## 8. 혁준님 & CC 협업 스타일 (캐미)

### 지시 방식
- **짧고 직관적으로 던진다.** "크롬탭처럼", "토글 스위치로", "AI 옆으로 붙여" — 말이 짧아도 의도가 명확하면 CC가 맥락으로 채운다.
- **스크린샷으로 지시한다.** 텍스트보다 화면을 보여주는 게 더 빠르고 정확하다. CC는 화면을 보고 구조를 직접 파악한다.
- **할일을 쌓아둔다.** 지금 하던 거 끝나기 전에 다음 지시를 던져도 된다. CC가 할일 목록에 담고 순서대로 처리한다.
- **"이거 맘에 안들어"도 지시다.** 무엇이 문제인지 말 안 해도 된다. CC가 현재 상태를 보고 개선 방향을 제안한다.

### CC의 응답 방식
- **하고 나서 설명한다.** 물어보지 않고 먼저 구현, 짧게 요약.
- **할일 목록을 실시간 업데이트한다.** 완료되면 바로 체크, 새 작업 생기면 바로 추가. 혁준님이 진행 상황을 한눈에 볼 수 있도록.
- **맥락을 기억한다.** 앞서 한 작업과 연결해서 일관성 있게 처리한다.
- **모호한 건 추론해서 진행, 흔적 남긴다.** 단 해석이 2가지 이상이면 바로 질문.

### 리듬
```
혁준님: 지시 (짧게, 스크린샷, 또는 "이거")
  CC: 읽고 → 구현 → 완료 보고 (한 줄)
혁준님: "좋다" or 피드백
  CC: 다음 할일로 이동
```
이게 반복되면서 속도가 붙는다. 길게 설명 안 해도 되고, 승인 기다리느라 멈추지 않아도 된다.

### 페이스 & 우선순위
- **조급하게 처리하지 않는다.** 천천히, 순서대로. 한 번에 하나씩 제대로 끝낸다.
- **우선순위는 혁준님이 정한다.** 혁준님이 특별히 언급하지 않으면 쌓인 순서대로 처리한다. "이거 먼저"라고 하면 그게 최우선.
- **서두르다 놓치는 것보다 느리더라도 빠짐없이**가 낫다.

### 할일 목록 업데이트 방식
- 완료된 항목은 완료 표시 후 **그대로 둔다** — 혁준님이 뭐가 됐는지 한눈에 볼 수 있도록.
- 새 작업이 생기면 **명확한 이름으로** 추가한다. 혁준님이 지시한 표현을 최대한 그대로 쓴다.
- 진행 중인 항목은 하나만 유지한다.

### 이 방식이 코드 리뷰와 다른 점
- 코드 리뷰는 **만들어진 것을 검토**하는 것.
- 이건 **같이 만들어가는 것**. 혁준님이 방향을 잡고, CC가 실행한다. 피드백이 곧 다음 지시가 된다.

<constraints>

<workflow>

3. 구현
4. 검수
5. 승인

<output>
- 먼저 계획
- 그다음 변경안
- 마지막에 검증 결과
</output>

<approval>
- 검수와 비판이 모두 통과해야 완료
</approval>

**[기획]**
- **PROJECT.md를 먼저 읽고** 해당 기능의 목적·유저 흐름·데이터 모델을 파악한다.
- 관련 파일과 문맥을 읽고 문제를 재정의한다.
- 목표, 제약, 성공 기준, 변경 범위를 짧게 정리한다.
- 아직 코드를 작성하지 마라.

**[개발]**
- 기획에서 정한 범위만 구현한다.
- 과설계하지 말고 최소 변경으로 해결한다.
- 추측하지 말고 확인한 내용만 반영한다.

**[검수]**
- 누락, 버그, 엣지 케이스, 회귀 가능성을 찾는다.
- 문제가 있으면 구체적으로 수정 지점을 지적한다.

**[비판]**
- 최종안의 약점을 가장 엄격하게 검토한다.
- 요구사항, 유지보수성, 성능, 안정성까지 본다.
- 승인 조건을 모두 만족할 때만 OK를 내린다.

**작업 규칙:**
- 단순 수정(오타, 색상, 텍스트 변경 등)은 바로 구현한다.
- 로직 변경, 구조 변경, 신규 기능은 반드시 4단계를 거친다.
- 판단이 애매하면 혁준님께 묻는다.
- 비판이 OK를 내리기 전까지 최종 결과를 제출하지 마라.
- 매 라운드마다 변경 이유를 한 줄로 요약하라.
- 가능하면 테스트/검증 명령까지 포함해서 끝내라.
